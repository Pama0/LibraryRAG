# 路由架构思路：有状态对话 agent + 确定性主题 workflow（设计 note）

> 状态：**方向探讨，未落地**。记问题、诊断、目标架构、纪律红线与取舍，给将来真要改时一个锚。
> 起因：2026-06-19 连续几个误判 + 一轮设计讨论，暴露了当前"单句分类-路由"范式的结构性问题。

## 1. 问题：枚举陷阱

当前 QA 预处理把 query 判进 **6 选 1 的扁平 enum**
（`retrievable / pending_split / ambiguous / missing_info / other / out_of_scope`），
上游另有 IntentRouter 的 `qa / study_plan / chitchat`。

每出现一类新失败，反射动作是"给提示词加一个类"。这条路**不收敛**：类数无界增长、每个边界都是 LLM 新的误判面、总有某格沦为"接不住的都丢这"的兜底垃圾桶、prompt 越长缓存越差类间越渗漏。

### 戳破"加类"反射的证据

| 问题 | 实际应归 | 被误判成 | 性质 |
|---|---|---|---|
| 「我完全不会mysql，给我讲懂mysql的概念」 | retrievable / pending_split（MySQL 在库） | out_of_scope | 现有类**误分类**（judge 拿"入门 vs 高级"当相关性轴） |
| 「为什么回答不了，你不是有MySQL书吗」 | chitchat（已存在的类） | out_of_scope | 路由**漏判** + out_of_scope 当兜底 |

按"一 bug 一类"去做，这两次加类**都是错的**。真正病灶是 out_of_scope 被当成"qa 接不住的一切"的垃圾桶。

## 2. 诊断之一：扁平 enum 压扁了正交的"主题轴"

6 个类其实横跨几个**互相独立**的判断轴，被硬拍成一维：

| 轴 | 在问什么 | 本质 | 当前对应类 |
|---|---|---|---|
| 意图 | 这是不是内容问题 | 意图分类 | qa vs chitchat/study_plan |
| **scope** | 主题在不在库 | **检索证据**（可测布尔） | out_of_scope = "否"分支 |
| **完整性** | 问句够不够具体、要不要反问 | query 完整性 | missing_info = "需澄清"分支 |
| **结构** | 单轮 / 并行扇出 / 多跳 | 分解需求 | retrievable / pending_split / other / ambiguous |

拍平成一维 → 类数 = 各轴笛卡尔积 → 组合爆炸 + 兜底格泄漏。out_of_scope 同时吃了"深度不匹配"和"元问题"——**不同轴塞进同一格**，正是泄漏根源。

> 观察：真正"多取值"的只有**结构**轴；scope/完整性/意图 都是小布尔/三态，被混进来才显得类多。

## 3. 诊断之二：还有一维，单句分类根本接不住

「你逗我呢」「为什么答不了」「那个再细讲讲」「回到刚才那个锁」——这些的意思**不在这句话里，在它和会话的关系里**。不管切几个正交的**主题轴**，对**单句**做分类都接不住，因为信号在**会话状态**里。它其实是一条**"自包含性"边界**：这一轮能不能脱离上下文独立理解？分两个子情形：

| 子情形 | 例 | 需要的状态 | 往回多远 |
|---|---|---|---|
| repair / 反馈 | "你逗我呢""为什么答不了" | 上一轮的**结果**（成/败） | 相邻 |
| 话题回指 / 重提 | "那个再讲讲""回到最开始的X" | 早轮的**内容** | 任意早轮 |

两点关键：
- 触发点不是"老话题"，是"**自不自包含**"。"MySQL聚簇索引怎么回事"哪怕几轮前聊过也是自包含 → 当普通新问题正常走主题流程，无需特殊处理。难的只有**非自包含 + 指代落很早**。
- **回指/repair 解析失败 → 优雅落回已有的 `missing_info`/clarify**（"你说的'那个'指前面哪个？"），不瞎猜。失败不需要新机制。

## 4. 目标架构：两层

把"对话表层"与"主题决策"彻底分开——一层用 agent 的**推理 + 记忆**吸收无界的对话长尾，一层用 workflow 的**确定性 + 证据**做有界的主题决策。

```
┌─ 有状态对话 agent（完整记忆 + 有界工具）──────────────────────┐
│  职责：对话表层（repair / 闲聊 / 回指消解 / 自包含化）          │
│  · 是对话/反馈/元问题   → 自己应付（不检索、不答内容）          │
│  · 是内容问题           → 拿记忆把它消解成【自包含 query】→ 下沉 │
└───────────────────────────────┬──────────────────────────────┘
                       【干净自包含 query】
┌───────────────────────────────▼──────────────────────────────┐
│  确定性主题 workflow（被 agent 当引擎调用）                     │
│  scope（锚检索证据）→ 完整性（够不够，否则 clarify）→ 结构      │
│  （单轮 / 并行扇出 / 多跳）→ 检索 → 有据答案                    │
└───────────────────────────────────────────────────────────────┘
```

- 我们纠结的"自包含性边界"**不再是第五个 gate**，而是**整个前置 agent 的职责**。
- 四个**主题轴**原封不动留在 workflow 里（scope 仍应锚在**检索证据**——召回片段主体实体 vs 问题主体实体是否一致——而非 LLM 凭感觉挑类，这是单点杠杆最高的一处）。
- 一个负责"这句话和对话什么关系"，一个负责"这个 query 关于什么、怎么查"，各司其职。

## 5. 这是把现状"翻转"

现在：**workflow(IntentRouter) 在前、agent 当 `other` 叶子**。
目标：**agent 当门口/编排、workflow 当它调用的引擎**。
与 CLAUDE.md "workflow 和 agent 是可组合的积木、不是二选一" 一致——只是组合方向反过来。

## 6. 三条纪律红线（破了就退化成裸 agent：贵 + 幻觉）

刚测过裸自主 agent ≈ 47s/条、12615 tokens/条（workflow 的 ~10×/100×）且会幻觉（见 [EVAL_ITERATION.md](EVAL_ITERATION.md)）。前置 agent 必须焊死：

1. **绝不自己答内容问题**：承载知识的提问一律转 workflow，agent 只**转述/格式化**有据答案。否则破"防幻觉 + 大模型知识与 RAG 源隔离"。
2. **不自己做检索**：检索是 workflow 的事。它要是自己反复 book_search，就是那个 47s/12k token 的裸 agent。
3. **动作是有界工具契约**，非自由发挥——大致 4 个出口：`答对话 / 反问澄清 / dispatch_to_qa(clean_query) / dispatch_to_study_plan`。门口收窄成几个出口才可观测、可测。

## 7. 取舍 / 边界（诚实说）

- **记忆别用 raw 全历史**——token 随轮数爆。用**压缩 running summary + 最近几轮**，接项目已有的 context-compression 方向。回指能力上限 = agent 能看多长的压缩记忆，是个可调旋钮。
- **可观测性下降**：门口从 classifier（可测、可复现）变成 agent 推理决策，不如分类好测。
- **评测要从单轮升到多轮**：golden 得加多轮 fixture；要测的不再只是"分类准不准"，而是"**对的 clean_query 有没有进 workflow**" + 对话轮处理是否得当。这是重构里**最大的工作量与不确定性**。
- **别滑进完整对话管理**（跨多轮目标追踪、对话状态机）——大坑。最小高价值切片：agent 持压缩记忆 + 做自包含化/repair + 把内容下沉 workflow。够杀掉观测到的螺旋，不必先建对话管理器。

## 8. 与现状的迁移映射

| 现状 | 去哪 | 备注 |
|---|---|---|
| IntentRouter（qa/chitchat + 指代消解） | **并入前置 agent** | 指代消解本就是它的活，agent 有完整记忆后做得更稳（长窗口回指） |
| out_of_scope | workflow 的 **scope** 轴 | 改成检索证据判定，别当兜底；非技术问题属对话层、不进这里 |
| missing_info | workflow 的 **完整性** 轴 | 收窄回本义；兼作回指/repair 解析失败的落点 |
| retrievable / pending_split / other / ambiguous | workflow 的 **结构** 轴 | "怎么分解检索"一个轴的四种取值，保留 |
| QaAgent（现 `other` 叶子） | 仍是 workflow 内的高难度分支 | 与前置编排 agent 是两个东西，别混 |

## 9. 开放问题（待想清再动）

- **scope 证据判定**怎么实现？纯阈值易被 frontmatter 噪声骗（见 probe 召回噪声讨论）；实体匹配需可靠实体抽取。大概率仍要 LLM，但**喂它证据判布尔**，比 6 类里挑稳。

- **多轮评测口径**：怎么造多轮 golden、怎么度量"clean_query 下沉是否正确"，工作量另估。
- **成本核算**：前置 agent 调用替代 IntentRouter 调用，内容轮 net 持平、对话轮更省（不进 workflow）——但需实测确认没引入额外多轮 LLM 调用。
