# 优化点记录

## 优化1：条文专用切分器（5.17）

**问题**：默认 SentenceSplitter 按固定字数切分，会将不同条文混在一起，导致检索命中错误条文。

**解决**：自定义 `ArticleSplitter(NodeParser)`，按"第X条"边界切分，每条为独立节点，元数据写入章节和条文信息。

**改动文件**：`core/rag/parser.py`

---

## 优化2：元数据 ref_doc_id 修复（5.17）

**问题**：`ArticleSplitter` 手动创建 TextNode 时未设置 `relationships`，导致 ChromaDB 中 `ref_doc_id`、`doc_id`、`document_id` 均为字符串 `"None"`。

**解决**：创建 TextNode 时添加 `NodeRelationship.SOURCE` 关系，指向源节点。

**改动文件**：`core/rag/parser.py`

---

## 优化3：元数据过滤 — AutoRetriever（5.17~5.18）

**问题**：纯向量检索无法处理结构化查询（如"第十一条是什么"最高相似度仅0.27，检索到错误章节）。

**方案演进**：

1. ~~正则提取~~ → 放弃。正则只能枚举查询模式，无法处理自然语言描述（"关于培训的规定"）、范围查询（"第三条到第十条"）、多法规查询等，后期维护成本无限膨胀。

2. **定制 AutoRetriever** → 最终方案。让 LLM 理解用户意图，输出结构化过滤条件。

**实现细节**：

- 新增 `article_no_int` 元数据字段（整数），支持范围查询（`article_no` 为中文数字无法做 GT/LT 比较）
- 定义 `VectorStoreInfo`，描述 5 个可过滤字段的含义，供 LLM 理解
- 定制中文法律场景 prompt，替换默认英文示例（Taylor Swift → 法律条文）
- AutoRetriever 提取失败时兜底退回纯向量检索

**改动文件**：

| 文件 | 改动 |
|------|------|
| `core/rag/parser.py` | 新增 `cn_num_to_int()`，元数据增加 `article_no_int` |
| `core/rag/vector_store_info.py` | 新建，定义 VectorStoreInfo |
| `core/rag/auto_retriever_prompt.py` | 新建，中文法律场景 prompt |
| `core/workflow/simple_rag.py` | 用 AutoRetriever 替换正则方案 |

**效果验证**：

| 查询 | LLM 提取结果 | 效果 |
|------|-------------|------|
| "第十一条是什么" | `article_no == "十一"` | 精准命中第十一条 |
| "第三条到第五条的内容" | `article_no_int >= 3 AND <= 5` | 返回第3~5条 |
| "安全生产有什么要求" | `file_name == "中国安全生产法.docx"` + query="安全生产" | 按法规+语义检索 |
